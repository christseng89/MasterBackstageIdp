## Running the Python Apps Locally

Python version is managed with `pyenv`; dependencies are managed with `uv`.

```bash
cd python-app
pyenv local 3.12.10
uv sync
uv pip install -r requirements.txt

python src/app.py       # Flask server on 0.0.0.0:5000
```

Note: `uv run main.py` runs a stub that only prints "Hello from python-app!" â€” it does not start the Flask server.

API endpoints exposed by `src/app.py`:
- `GET /` â€” returns "Hello World!"
- `GET /api/v1/info` â€” returns time, hostname, and a message

There are no automated tests in this repository.

## Architecture Overview

### Two GitOps Patterns

#### python-app â€” Single Environment

Defined in `.github/workflows/cicd.yaml`; triggered on pushes to `python-app/src/**` on `main`:

1. **CI** (`ubuntu-latest`): builds and pushes a **multi-arch** (linux/amd64 + linux/arm64) Docker image to Docker Hub tagged with the first 6 chars of the commit SHA (`christseng89/python-app:<commit_id>`).
2. **CD** (`self-hosted` ARC runner): uses `yq` to update `image.tag` in `python-app/charts/python-app/values.yaml`, commits back to `main` using `GH_PAT`, then calls `argocd app sync python-app`.

`python-app/charts/python-app/values.yaml` is the single source of truth for the deployed version. ArgoCD app: `python-app`.

#### python-app4 â€” Multi-Environment (dev / staging / prod)

> **Note:** The `python-app4` app directory is not checked out locally â€” this section documents the multi-environment GitOps pattern it demonstrates.

Two separate workflows in `python-app4/.github/workflows/`:

- **`python-app4-cicd.yaml`** â€” triggered on `python-app4/src/**` changes: runs CI (build + push image), then CD auto-deploys to **dev** by updating `charts/python-app4/values-dev.yaml` using `GITHUB_TOKEN` + `EndBug/add-and-commit@v9`.
- **`python-app4-cd.yaml`** â€” triggered when `values-staging.yaml` or `values-prod.yaml` change (manual promotion flow): detects which values file changed, validates the image tag is non-empty, then syncs the corresponding ArgoCD app (`python-app4-staging` or `python-app4-prod`).

ArgoCD apps: `python-app4-dev` (namespace `python-app4-dev`), `python-app4-staging` (namespace `python-app4-staging`), `python-app4-prod` (namespace `python-app4-prod`). Each environment gets its own per-app namespace so RBAC, quotas, and network policies stay scoped per service.

**Promotion model:** edit the target `values-{env}.yaml` with a new `image.tag` and push to `main` â€” the CD workflow fires automatically.

### Helm Chart vs Raw K8s Manifests

Each app has both:
- `<app>/charts/<app>/` â€” Helm chart; what ArgoCD reads; `values.yaml` is the base, `values-{env}.yaml` per-env overrides
- `<app>/k8s/` â€” raw Kubernetes manifests for direct `kubectl apply` use

Shared infrastructure charts:
- `python-app/charts/argocd/values-argo.yaml` â€” values override for deploying ArgoCD itself via Helm (domain `argocd.test.com`)
- `python-app/charts/nginx/values-nginx.yaml` â€” Nginx Ingress Controller overrides

### ARC Runner

The original `python-app/runnerdeployment.yaml` deploys a runner in the `python-app` namespace. The scaffolder template uses a different, multi-app design:

- **Shared namespace:** All scaffolded-app runners run in `github-runners` (not a per-app namespace).
- **Per-app ServiceAccount:** `<app_name>-self-hosted-runner` in `github-runners` â€” isolates kubectl permissions per app.
- **Shared ClusterRole `arc-runner-reader`:** read-only access to pods, events, deployments, replicasets â€” applied once per cluster.
- **Per-env RoleBinding:** lives inside each app's Helm chart (`charts/<app>/templates/runner-rolebinding.yaml`), applied by ArgoCD on first sync of each env. This means the runner gains access to `<app>-dev`, `<app>-staging`, `<app>-prod` namespaces automatically as each env is deployed â€” no manual RBAC setup.

The runner needs in-cluster DNS to reach `argocd-server.argocd.svc.cluster.local`. The `github-runners` namespace is kept outside ArgoCD management deliberately: ArgoCD `auto-prune` would delete the runner pod (it's not in the Helm chart), causing an ARC reconcile loop.

### Backstage Integration

- `<app>/catalog-info.yaml` â€” registers the app as a Backstage `Component`; `python-app` also registers an `API` (type: openapi) pointing at `python-app/openapi.yaml`
- `<app>/mkdocs.yaml` + `<app>/docs/` â€” TechDocs source; `backstage.io/techdocs-ref: dir:.` points Backstage at the MkDocs config
- `backstage-app/backstage/catalog/entities/` â€” `groups.yaml` + `users.yaml` loaded in Docker/production deployments (not in base `app-config.yaml`)
- TechDocs is configured as `builder: local` â€” Backstage backend runs mkdocs on demand
- MCP Actions are enabled in the backend (`pluginSources: auth, catalog, scaffolder`) via `app-config.yaml`

### Scaffolder Templates

Two templates live under `backstage-app/templates/`; both are registered in `app-config.local.yaml` as local file sources mounted at `/app/templates/` inside the container.

Both templates use the same multi-environment GitOps skeleton (like `python-app4`) and produce the same four workflow files, Helm chart layout, ArgoCD app auto-creation, and `setup.sh` bootstrap. The difference is in the Python application itself.

#### python-app â€” Single-version microservice

`backstage-app/templates/python-app/template.yaml` provisions a basic Flask microservice with a single `/api/v1` surface. Skeleton at `backstage-app/templates/python-app/template/`.

Generated repo gets:
- `.github/workflows/<app_name>-cicd.yaml` â€” CI (build + push) + auto-deploy to **dev**
- `.github/workflows/<app_name>-staging-cd.yaml` â€” triggered when `values-staging.yaml` changes; promotes to **staging**
- `.github/workflows/<app_name>-prod-cd.yaml` â€” triggered when `values-prod.yaml` changes; promotes to **prod**
- `.github/workflows/mirror-cli-binaries.yaml` â€” mirrors argocd/yq/kubectl binaries to Docker Hub (manual trigger)
- Three ArgoCD apps: `<app_name>-dev`, `<app_name>-staging`, `<app_name>-prod` (each in its own namespace)

#### python-app-multi-apis â€” Versioned API microservice

`backstage-app/templates/python-app-multi-apis/template.yaml` provisions a Flask microservice that demonstrates the **additive multi-version API pattern**: one image simultaneously serves `/api/v1`, `/api/v2`, and `/api/v3` with distinct lifecycle states. Skeleton at `backstage-app/templates/python-app-multi-apis/template/`.

Key differences from python-app:
- **Three Backstage API entities** (type: openapi) registered alongside the Component â€” each with its own `lifecycle` value (`deprecated`/`production`/`experimental`) and inline OpenAPI spec in `catalog-info.yaml`
- **Flask blueprints per version** (`src/api_v1.py`, `api_v2.py`, `api_v3.py`); register/unregister blueprints in `app.py` â€” no per-version Docker builds
- **RFC 9745/8594 deprecation headers** injected by `src/deprecated_headers.py` after-request middleware on any path in `DEPRECATED_PREFIXES` (currently `/api/v1/`)
- **`GET /version` endpoint** (`src/version_endpoint.py`) returns image tag, git SHA, supported/deprecated version lists, and Backstage entity refs; reads `IMAGE_TAG`, `GIT_SHA`, `APP_NAME`, `DEFAULT_API_VERSION` from env (injected by Helm)

API route summary:

| Version | Lifecycle | Endpoints |
|---------|-----------|-----------|
| v1 | deprecated (sunset 2025-12-31) | `/api/v1/info`, `/api/v1/healthz`, `/api/v1/hostname` |
| v2 | production | v1 routes + `/api/v2/echo`, `/api/v2/time` |
| v3 | experimental | v2 routes + `/api/v3/random`, `/api/v3/uptime`, `/api/v3/env` |

To deprecate a version: add its path prefix to `DEPRECATED_PREFIXES` in `deprecated_headers.py` and flip `lifecycle` in `catalog-info.yaml`. Actual code removal waits until the sunset date passes **and** traffic drops below 1% â€” at that point return HTTP 410 (not 404) before deleting the blueprint.

#### Post-scaffolding bootstrap (both templates)

Each generated repo includes `setup.sh`. Run once from the repo root after cloning:

```bash
# Create .env first with DOCKERHUB_USERNAME, DOCKERHUB_TOKEN, ARGOCD_PASSWORD, GITHUB_PAT
bash setup.sh                    # all steps
bash setup.sh --skip-mirror      # skip binary mirror (mirrors already exist on Docker Hub)
bash setup.sh --skip-cicd        # skip first workflow trigger
```

`setup.sh` handles: registering the ARC runner pod, setting all GitHub secrets/variables, triggering the mirror workflow, writing the three Windows hosts entries (`<app>-dev/staging/prod.test.com`), and triggering the first `cicd.yaml` run. Requires `kubectl` (docker-desktop context) and an authenticated `gh` CLI.

Staging and prod workflows **auto-create the ArgoCD app** on first run if it doesn't exist yet (`argocd app create --validate=false`), so no manual ArgoCD setup is needed. The `${{values.app_name}}` placeholder in skeleton directory names is replaced with the component ID at scaffold time.

### Cluster Infrastructure Setup

Install Nginx Ingress Controller and ArgoCD into the local Docker Desktop cluster:

```bash
# Nginx Ingress (exposes services on port 9080)
cd python-app
helm repo add ingress-nginx https://kubernetes.github.io/ingress-nginx && helm repo update
helm upgrade --install ingress-nginx ingress-nginx/ingress-nginx \
  -n ingress-nginx --create-namespace -f charts/nginx/values-nginx.yaml

# ArgoCD
helm repo add argo https://argoproj.github.io/argo-helm && helm repo update
helm upgrade --install argocd argo/argo-cd \
  -n argocd --create-namespace -f charts/argocd/values-argo.yaml

# Retrieve the initial admin password
kubectl -n argocd get secret argocd-initial-admin-secret \
  -o jsonpath="{.data.password}" | base64 -d
```

ArgoCD UI is then at `http://argocd.test.com:9080` (username `admin`).

**ArgoCD `python-app` application path must be `python-app/charts/python-app` exactly.** The CD job writes to `python-app/charts/python-app/values.yaml`; a different path means ArgoCD sees no diff and the pod never updates.

`mirror-cli-binaries.yaml` (exists in both `.github/workflows/` and `python-app4/.github/workflows/`) mirrors ArgoCD, yq, and kubectl binaries to Docker Hub (`christseng89/{argocd,yq,kubectl}-bin`) for fast in-cluster pulls. The CD jobs cache these binaries at `/tmp/{argocd,yq,kubectl}` keyed by version + arch.

## Critical Operational Notes

**Mirror workflow before version bumps:** When upgrading `ARGOCD_VERSION` or `YQ_VERSION`, run `.github/workflows/mirror-cli-binaries.yaml` (workflow_dispatch) with the new version *first*. Bumping the version in `cicd.yaml` before the mirror image exists causes the CD job to fail with `manifest for christseng89/argocd-bin:vX.Y.Z not found`.

**Avoid non-fast-forward push rejections:** The CD job commits `values.yaml` changes back to `main`. Configure git once locally so `git pull` uses rebase:

```bash
git config --global pull.rebase true
```
